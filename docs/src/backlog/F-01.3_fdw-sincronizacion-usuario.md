**Descripción:**
Como sistema backend,
Quiero copiar los datos básicos del usuario desde Keycloak a la tabla `registros_ceed` en el primer login,
Para disponer de un perfil local del usuario desde el momento en que accede al dashboard por primera vez.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Primer login — creación de registro):**
  **Dado que** un usuario se autentica en el dashboard por primera vez,
  **Cuando** el backend valida el JWT y detecta que no existe un registro previo para ese `email` en `registros_ceed`,
  **Entonces** crea un nuevo registro con los datos básicos obtenidos de Keycloak: `email`, `given_name`, `family_name` y el `sub` como identificador.

- [ ] **Escenario 2 (Logins posteriores — sin duplicación):**
  **Dado que** el usuario ya tiene un registro en `registros_ceed`,
  **Cuando** vuelve a iniciar sesión,
  **Entonces** no se crea un registro duplicado y el usuario accede directamente con su perfil existente.

- [ ] **Escenario 3 (FDW en producción):**
  **Dado que** en producción la BD de Laravel y la BD de Keycloak están en el mismo servidor PostgreSQL,
  **Cuando** se accede a los datos de Keycloak mediante FDW,
  **Entonces** la sincronización lee los datos directamente de la BD de Keycloak via Foreign Data Wrapper.

- [ ] **Escenario 4 (Entorno local sin FDW):**
  **Dado que** en entorno local el FDW no está activo,
  **Cuando** se ejecuta la sincronización en desarrollo,
  **Entonces** se usan los claims del JWT para poblar los datos básicos del usuario (sin FDW), usando tablas propias.

- [ ] **Requisito Técnico (Independencia post-sincronización):**
  Una vez creado el registro en `registros_ceed`, los datos son independientes de Keycloak. Las ediciones posteriores del perfil (IBAN, matrículas, etc.) se gestionan directamente en `registros_ceed` sin volver a sincronizar desde Keycloak.

---

**MoSCoW:** `Must`
**Sprint:** Sprint 1

**Notas:**

- FDW activo solo en producción (BD de Laravel y Keycloak en el mismo servidor PostgreSQL). En local se usan tablas propias.
- **Implementación FDW:** Se debe crear una migración o script SQL (`CREATE EXTENSION postgres_fdw; CREATE SERVER ...`) parametrizado vía `.env` para facilitar el setup del FDW en el entorno correspondiente.
- Los campos básicos provienen de Keycloak: `email`, `given_name`, `family_name`. El resto (IBAN, puesto, matrículas, etc.) son propios de `registros_ceed` y se gestionan independientemente.
- La sincronización ocurre una única vez: en el primer login. No hay sincronización continua.
- El campo `identidad_digital` en `registros_ceed` = `email` claim del JWT de Keycloak.
- Los roles existen en Keycloak pero no se implementan en esta fase; no se sincronizan.
