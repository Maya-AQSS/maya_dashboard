**Descripción:**
Como sistema RabbitMQ (productor externo),
Quiero llamar a un endpoint REST del dashboard para insertar notificaciones,
Para entregar mensajes a los usuarios sin que el dashboard tenga que consumir directamente de la cola.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Inserción exitosa con API key válida):**
  **Dado que** RabbitMQ envía un POST con una API key válida en la cabecera y un payload correcto,
  **Cuando** el backend recibe la petición,
  **Entonces** crea la notificación en la base de datos y devuelve `201 Created` con el ID de la notificación creada.

- [ ] **Escenario 2 (Rechazo por API key inválida):**
  **Dado que** la petición llega con API key incorrecta o ausente,
  **Cuando** el backend procesa la autenticación,
  **Entonces** devuelve `401 Unauthorized` sin crear ningún registro.

- [ ] **Escenario 3 (Idempotencia por `external_id`):**
  **Dado que** RabbitMQ reintenta el envío de una notificación ya procesada (mismo `external_id`),
  **Cuando** el backend recibe la petición duplicada,
  **Entonces** no crea un registro duplicado y devuelve `200 OK` o `409 Conflict` indicando que ya fue procesada.

- [ ] **Escenario 4 (Validación de payload):**
  **Dado que** el payload enviado por RabbitMQ no cumple el esquema mínimo requerido,
  **Cuando** el backend valida la petición,
  **Entonces** devuelve `422 Unprocessable Entity` con descripción de los campos inválidos.

- [ ] **Escenario 5 (Campos del payload):**
  **Dado que** se inserta una notificación válida,
  **Cuando** se persiste en la base de datos,
  **Entonces** el registro incluye: `title`, `body`, `type`, `priority` (normal/aviso/urgente), `user_id`, `action_url` (opcional), `expires_at` (opcional), `external_id` y `read_at` (null por defecto).

- [ ] **Requisito Técnico (API key única global):**
  La API key es única y global para todos los sistemas origen. Se configura vía variable de entorno. No se expone en logs ni en respuestas de error.

**MoSCoW:** `Must`
**Sprint:** Sprint 2

**Notas:**

- La API key se autentica vía cabecera HTTP (ej. `X-API-Key`). El mecanismo exacto es decisión de implementación.
- El endpoint está excluido del middleware JWT — usa exclusivamente API key.
- El esquema del payload se define en este proyecto (no hay contrato externo impuesto). Los campos propuestos en los criterios son la definición canónica.
- `priority` acepta: `normal`, `aviso`, `urgente`. Solo las urgentes dispararan WebSocket (F-03.3).
- La retención de notificaciones es ajustable al crear cada una. Sin expiración por defecto. No leídas: conservadas indefinidamente.
