**Descripción:**
Como desarrollador backend,
Quiero que Laravel valide los tokens JWT emitidos por Keycloak mediante el JWKS endpoint,
Para proteger todos los endpoints de la API y garantizar que solo usuarios autenticados acceden a los datos.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Token válido aceptado):**
  **Dado que** un cliente envía una petición HTTP con un JWT válido en la cabecera `Authorization: Bearer`,
  **Cuando** Laravel procesa la petición,
  **Entonces** valida la firma del token contra el JWKS endpoint de Keycloak y permite el acceso al endpoint solicitado.

- [ ] **Escenario 2 (Token inválido rechazado):**
  **Dado que** un cliente envía un JWT expirado, malformado o con firma inválida,
  **Cuando** Laravel procesa la petición,
  **Entonces** retorna una respuesta `401 Unauthorized` con mensaje de error apropiado, sin exponer detalles internos.

- [ ] **Escenario 3 (Claim de identidad):**
  **Dado que** el JWT es válido,
  **Cuando** Laravel extrae la identidad del usuario,
  **Entonces** utiliza el claim `email` del JWT como valor de `identidad_digital` para identificar al usuario en la base de datos.

- [ ] **Escenario 4 (Endpoint público excluido):**
  **Dado que** el endpoint de inserción de notificaciones usa API key (no JWT),
  **Cuando** se define el middleware de autenticación,
  **Entonces** ese endpoint está excluido del middleware JWT y usa su propio mecanismo de autenticación por API key.

- [ ] **Requisito Técnico (Resource Server puro):**
  Laravel actúa como resource server puro — solo valida tokens emitidos al cliente frontend. No emite tokens propios ni requiere cliente confidencial en Keycloak.

---

**MoSCoW:** `Must`
**Sprint:** Sprint 1

**Notas:**

- **Instalación requerida:** Instalar la librería mediante `composer require firebase/php-jwt`.
- **Implementación:** Crear un Custom Auth Guard en Laravel que utilice esta librería para validar los tokens en cada request protegida.
- La URL del JWKS endpoint de Keycloak se configura vía variable de entorno (`KEYCLOAK_JWKS_URI`).
- El backend NO tiene cliente confidencial en Keycloak — es resource server puro.
- Todos los endpoints de la API (excepto el de notificaciones con API key) deben estar protegidos por este middleware.
- Los roles en el token existen pero no se consumen en esta fase — se reserva para implementación futura de authorization.
