# [SECURITY] F-01.4 — Persistencia y Renovación de Sesión en Frontend

**Épica:** E-01 — Autenticación y Gestión de Sesión

## 🏷️ Categoría

**Categoría:** `Security`
**Impacta en:** Equipo Frontend

| --- | --- | --- |
| [F-01.1](F-01.1_oidc-frontend.md) | Provee el flujo OIDC inicial y los tokens (access_token + refresh_token) que esta feature gestiona | Sí |

**Prioridad:** `Must`
**Sprint:** Sprint 1

---

**Descripción:**
Como usuario del dashboard,
Quiero que mi sesión se mantenga activa mientras uso la aplicación y se renueve automáticamente antes de expirar,
Para no ser desconectado de forma inesperada durante mi jornada de trabajo.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Renovación automática del token):**
  **Dado que** el usuario tiene una sesión activa y el access_token está próximo a expirar,
  **Cuando** el frontend detecta la proximidad de expiración,
  **Entonces** solicita automáticamente un nuevo access_token usando el refresh_token, sin interrumpir la actividad del usuario.

- [ ] **Escenario 2 (Sesión expirada — redirigir a login):**
  **Dado que** el refresh_token ha expirado o ha sido revocado en Keycloak,
  **Cuando** el frontend intenta renovar el access_token,
  **Entonces** la sesión local se destruye y el usuario es redirigido al flujo de login de Keycloak.

- [ ] **Escenario 3 (Persistencia entre recargas):**
  **Dado que** el usuario tiene una sesión activa y recarga la página del navegador,
  **Cuando** la aplicación carga,
  **Entonces** detecta el token almacenado y restaura la sesión sin requerir un nuevo login, siempre que el token siga siendo válido.

- [ ] **Escenario 4 (Peticiones con token actualizado):**
  **Dado que** el token ha sido renovado,
  **Cuando** el frontend realiza peticiones a la API de Laravel,
  **Entonces** siempre usa el access_token más reciente en la cabecera `Authorization: Bearer`.

- [ ] **Requisito Técnico (Seguridad de almacenamiento):**
  Los tokens deben almacenarse de forma segura. No deben exponerse en URLs ni en logs del navegador.

---

**MoSCoW:** `Must`
**Sprint:** Sprint 1

**Notas:**

- La estrategia de almacenamiento del token (memoria, sessionStorage, etc.) es una decisión de implementación del equipo frontend. Considerar las implicaciones de seguridad de cada opción.
- La política de expiración del access_token y refresh_token se configura en Keycloak. El frontend debe adaptarse a esos valores.
- Al desconectarse, siempre invocar el endpoint de logout de Keycloak (implementado en F-01.1) para invalidar también la sesión en el servidor.
