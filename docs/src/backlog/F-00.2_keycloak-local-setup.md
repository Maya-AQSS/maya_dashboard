**Descripción:**
Como desarrollador del equipo,
Quiero disponer de una instancia Keycloak local preconfigurada con usuarios de test y claim mappers,
Para poder desarrollar y probar el flujo de autenticación OIDC sin depender del servidor Keycloak de producción.

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Instancia local arranca):**
  **Dado que** Docker Compose está corriendo,
  **Cuando** se accede a la consola de administración de Keycloak local,
  **Entonces** el realm configurado está disponible con los clientes `maya-frontend` (public) y la configuración de resource server para el backend.

- [ ] **Escenario 2 (Usuarios de test):**
  **Dado que** el realm local está configurado,
  **Cuando** se usa un usuario de test predefinido para iniciar sesión en el frontend,
  **Entonces** el flujo OIDC completa correctamente y se obtiene un token JWT válido.

- [ ] **Escenario 3 (Claim mappers):**
  **Dado que** el realm local está activo,
  **Cuando** se decodifica el JWT obtenido en un login de prueba,
  **Entonces** el token incluye los claims: `email`, `given_name`, `family_name` y el identificador de usuario (sub).

- [ ] **Escenario 4 (Realm exportable):**
  **Dado que** la configuración del realm está completa,
  **Cuando** se exporta la configuración como archivo JSON,
  **Entonces** existe un archivo de realm export en el repositorio (ej. `backend/keycloak/realm-export.json`) que permite reproducir la configuración con un único comando.

- [ ] **Requisito Técnico (Entorno local únicamente):**
  Esta configuración es exclusiva del entorno de desarrollo local. En producción se usa el servidor Keycloak externo ya existente (ver `https://github.com/Maya-AQSS/IaC.git`).

**Notas:**

- El realm de producción ya existe y está configurado en `github.com/Maya-AQSS/IaC.git`. Este backlog crea solo la instancia local para desarrollo.
- Los claim mappers que deben añadirse: `email`, `given_name`, `family_name`. Confirmar con el equipo de infra si ya están configurados en el realm de producción.
- Redirect URI local asumida: `http://localhost:5173`. Parametrizar por entorno.
- Cliente frontend: tipo `public`, Authorization Code + PKCE, sin client secret.
- Backend: resource server puro — no requiere cliente confidencial propio, solo valida tokens vía JWKS endpoint.
- Referencia Docker Compose Keycloak: `github.com/aoltra/odoodock`.
