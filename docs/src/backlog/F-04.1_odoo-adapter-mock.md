**Descripción:**
Como desarrollador backend,
Quiero disponer de una capa de abstracción para la API de Odoo con un mock funcional que simule las respuestas,
Para poder desarrollar y probar el módulo de fichajes sin depender de la disponibilidad de la API real de Odoo (disponible en junio 2026).

**Criterios de Aceptación (Checklist):**

- [ ] **Escenario 1 (Interface de abstracción definida):**
  **Dado que** el módulo de fichajes necesita consultar y registrar fichajes,
  **Cuando** se diseña la arquitectura,
  **Entonces** existe una interface (contrato) que define las operaciones del adapter: fichar entrada, fichar salida, obtener historial de fichajes del usuario.

- [ ] **Escenario 2 (Mock funcional devuelve datos simulados):**
  **Dado que** la API real de Odoo no está disponible,
  **Cuando** el sistema ejecuta cualquier operación de fichaje en el entorno local,
  **Entonces** el mock devuelve datos simulados coherentes (registros de entrada/salida con timestamps) que permiten desarrollar y probar la lógica completa.

- [ ] **Escenario 3 (Sustitución sin modificar lógica de negocio):**
  **Dado que** la API real de Odoo estará disponible en junio 2026,
  **Cuando** se implemente la integración real,
  **Entonces** se crea una nueva implementación concreta del adapter sin modificar ningún servicio, controlador o lógica de negocio existente.

- [ ] **Escenario 4 (Configuración por entorno):**
  **Dado que** el equipo quiere controlar qué implementación usar,
  **Cuando** se configura la variable de entorno correspondiente,
  **Entonces** el sistema usa el mock o la implementación real según la configuración, sin cambios en el código.

- [ ] **Requisito Técnico (Sin copia local del histórico):**
  El histórico de fichajes siempre se consulta desde Odoo (mock o real). No se almacena copia local en la BD del dashboard.

---

**MoSCoW:** `Must`
**Sprint:** Sprint 2

**Notas:**

- La API de Odoo estará disponible en junio 2026. El mock es obligatorio hasta entonces.
- Solo fichajes personales del usuario. Sin vistas de equipo ni de managers en esta fase.
- El histórico siempre viene de Odoo (sin copia local en BD de Laravel).
- El adapter pattern garantiza el desacoplamiento: la lógica de negocio en controladores y servicios solo conoce la interface, no la implementación concreta.
- El binding entre interface e implementación se configura en el Service Container de Laravel.
